# CyberGuard — Architecture

## Stack

| Layer | Technology | Notes |
|---|---|---|
| Frontend | Vanilla HTML5 / CSS3 / JavaScript (ES6+) | Zero build step, no npm dependencies |
| Auth | Supabase Auth | Email/password, OTP verification, password reset |
| Database | Supabase (PostgreSQL) | Profiles, game sessions, leaderboard, RLS |
| Hosting | Cloudflare Pages | Auto-deploy from GitHub `main`; build script generates config |
| Fonts | Bunny Fonts | Nunito · Inter · Rajdhani · Fjalla One — GDPR-friendly CDN |
| Animations | GSAP 3.12 | Screen transitions, XP burst, confetti |

### Script Load Order (index.html)

Scripts are loaded synchronously (no `async`/`defer`) to guarantee constant availability:

```
1. https://cdn.jsdelivr.net/npm/@supabase/supabase-js   → window.supabase
2. https://cdn.jsdelivr.net/npm/gsap                    → GSAP
3. js/config.js         → SUPABASE_URL, SUPABASE_ANON, AGE_GROUPS, REGIONS, BADGES, db
4. js/scenarios.js      → MISSIONS{} base object (kids/teens/adults/seniors arrays)
5. js/new_missions_kids.js      → pushes into MISSIONS.kids[]
6. js/new_missions_teens.js     → pushes into MISSIONS.teens[]
7. js/new_missions_adults.js    → pushes into MISSIONS.adults[]
8. js/scenarios-asean.js        → pushes regional missions into all MISSIONS arrays
9. js/scenarios-apac.js         → pushes regional missions into all MISSIONS arrays
10. js/app.js           → App state, auth, game engine, UI
```

**Important:** `config.js` declares `const db` at the **bottom** of the file, after all constants. This ensures `REGIONS`, `AGE_GROUPS`, etc. are always defined even if the Supabase CDN fails to load.

## Build System

Cloudflare Pages runs `build.sh` on each deploy. It uses Node.js to read `js/config.example.js`, substitute the `SUPABASE_URL` and `SUPABASE_ANON` placeholders with environment variables, and write `js/config.js`.

**Rule:** Always modify `js/config.example.js` for any changes to config constants. Edits to `js/config.js` are overwritten on the next deploy.

Required Cloudflare Pages environment variables:
- `SUPABASE_URL`
- `SUPABASE_ANON`

## Database Schema

```
profiles        — User profile: username, age_group (DB key), avatar, level, hobby,
                  intro_completed, foundational_completed, total_score, missions_completed,
                  badges (text[]), created_at, updated_at

game_sessions   — Mission play-through: user_id FK, age_group, mission_id, mission_type,
                  score, max_score, percentage (computed), completed, time_taken, created_at

level_attempts  — Per-question tracking: session_id FK, user_id FK, question_id,
                  correct, points_earned, created_at

leaderboard     — VIEW: RANK() OVER (PARTITION BY age_group ORDER BY total_score DESC)
                  + global ranking; read-only
```

### Key Design Decisions

- **DB keys unchanged** — `age_group` column stores `kids/teens/adults/seniors` (original values). UI labels were updated to Beginner/Intermediate/Advanced/Specialist without a schema migration.
- **RLS (Row Level Security)** — Users read/write only their own sessions. Leaderboard is read-only for all authenticated users. Admin users bypass via `is_admin()` function.
- **Score trigger** — PostgreSQL trigger on `game_sessions` auto-updates `profiles.total_score` when `completed = TRUE`, preventing frontend score manipulation.
- **Leaderboard VIEW** — Computed server-side; no client-side sorting needed.

## App Architecture

Single Page Application using the **screen-swap pattern**:
- All screens exist in the DOM simultaneously as `<div id="screen-*" class="screen">`
- Only one screen has `class="screen active"` at a time
- `showScreen(id)` handles all transitions (adds/removes `.active`)
- Supabase `onAuthStateChange` drives top-level screen routing

### Auth Flow

```
Page load → initAuth()
  ├── hash has #type=recovery → App.isRecovery = true → showScreen('screen-reset')
  ├── getSession() returns session → loadProfile() → renderDashboard() → screen-dashboard
  └── no session → showScreen('screen-landing')

onAuthStateChange:
  ├── PASSWORD_RECOVERY → App.isRecovery = true → screen-reset
  ├── USER_UPDATED → no-op (handled by calling function)
  ├── App.isRecovery = true → stay on screen-reset (suppress SIGNED_IN during reset flow)
  ├── SIGNED_IN (normal) → loadProfile()
  └── SIGNED_OUT → screen-landing
```

### Game Engine Flow

```
startMission()
  → renderMissionBriefing()
  → launchMission()
       ├── quiz:          renderQuiz() → selectQuizAnswer() → nextQuestion() → completeMission()
       ├── spot-threat:   renderSpotThreat() → handleThreatClick() → submitSpotThreat() → completeMission()
       └── decision-tree: renderDecisionTree() → chooseDtOption() → (recurse until leaf) → completeMission()
                                                           ↓
                                              DB: insert game_session + level_attempts
                                              trigger: update profiles.total_score
                                              awardBadges() → renderMissionComplete()
```

### XP Formula

```
effectiveXP = baseXP × ageGroupMultiplier × levelMultiplier

ageGroupMultiplier: based on difficulty delta between player's track and mission's track
  same track = ×1.0 | +1 harder = ×1.5 | +2 harder = ×2.0 | -1 easier = ×0.75 | -2 easier = ×0.5

levelMultiplier: based on delta between player's difficulty setting and mission's difficulty
  same = ×1.0 | harder = ×1.5/×2.0 | easier = ×0.75/×0.5
```

## Theme System

Three dark-only visual themes driven by `data-theme` attribute on `<html>`:

| Theme | Name in UI | Key Design |
|---|---|---|
| `teen` | Default | Near-black bg `#050A14`, purple-cyan neon, Rajdhani font |
| `kiddy` | Arcade | Warm dark bg, coral/yellow accent, rounded borders |
| `crimson` | Crimson | Deep dark bg `#0C0408`, red accent `#FF3040` |

Age-group structural themes (`kids`/`teens`/`adults`/`seniors`) are separate — they control font size, tap target size, and density. Both systems coexist:
- `data-theme` on `<html>` = visual colour scheme (user-choosable)
- `data-theme` on `<body>` = structural age-group layout (auto-applied on login)

Dark mode is locked (`data-mode="dark"` on `<body>`). Light/system toggle removed in v1.6.
